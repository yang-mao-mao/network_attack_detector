# 特征检测集成说明

## 1. 目的

`SignatureDetectionService` 是成员 C 提供给主程序、GUI 和抓包模块的统一控制层。它把规则加载、校验、内存管理、特征匹配和 `DetectionResult` 输出组合为一个稳定接口，同时不依赖 PyQt6 或 Scapy。

文件位置：

```text
src/detection/signature_service.py
```

调用关系：

```text
成员 E GUI ───────────────┐
                          ▼
成员 A 主程序 ──> SignatureDetectionService
                          │
成员 B PacketInfo ────────┤
                          ▼
             RuleLoader / RuleManager
                          │
                          ▼
                  SignatureEngine
                          │
                          ▼
                   DetectionResult
```

## 2. 初始化和规则加载

直接加载默认规则：

```python
from src.detection.signature_service import SignatureDetectionService

service = SignatureDetectionService("data/rules/signature_rules.csv")
```

也可以先创建空服务，再加载用户规则：

```python
service = SignatureDetectionService()
rules = service.load_rules("data/rules/user_rules1.csv")
```

规则文件会经过 `RuleLoader` 和 `RuleValidator` 的完整检查。加载失败会抛出 `RuleError`，不会激活部分规则。

## 3. 重新加载

重新加载上一次成功选择的文件：

```python
service.reload_rules()
```

加载另一个文件：

```python
service.reload_rules("data/rules/user_rules2.csv")
```

只有新文件完整加载成功后，服务才会替换活动规则并重建 `SignatureEngine`。如果失败，原规则、原引擎和原文件路径继续有效。

## 4. 数据包检测

成员 B 或主程序向服务传入标准 `PacketInfo`：

```python
result = service.detect(packet)
```

返回类型：

```python
DetectionResult
```

GUI 可以直接调用：

```python
feature_window.result_panel.add_result(result)
```

结果包含：

- `packet_id`
- `matched`
- `alerts`
- `engine_name="SignatureEngine"`
- `cost_ms`

## 5. 内存规则管理

```python
service.list_rules()
service.get_rule(rule_id)
service.add_rule(rule)
service.update_rule(rule)
service.remove_rule(rule_id)
service.enable_rule(rule_id)
service.disable_rule(rule_id)
service.replace_rules(rules)
```

结构性修改（新增、删除、更新、批量替换）会重建特征引擎，从而刷新正则缓存和数据包去重状态。启用和禁用只改变共享规则对象，可从下一个数据包开始生效。

内存规则修改不会自动写回 CSV。持久化仍由 GUI 或成员 A 的存储层负责。

## 6. GUI 接入流程

成员 E 的特征检测页面已经提供以下入口：

```text
RulePanel.get_enabled_rule_ids()
RulePanel.get_all_rules()
ResultPanel.add_result(result)
detection_started
detection_stopped
```

推荐接入流程：

```text
用户打开或保存规则文件
→ service.load_rules()/reload_rules()
→ 捕获 RuleError 并使用 QMessageBox 显示
→ 用户点击开始检测
→ 接收成员 B 产生的 PacketInfo
→ service.detect(packet)
→ ResultPanel.add_result(result)
```

规则复选框变化时调用：

```python
if checked:
    service.enable_rule(rule_id)
else:
    service.disable_rule(rule_id)
```

## 7. 线程约定

服务使用可重入锁保护规则变更和检测调用，可以从抓包工作线程调用。但 PyQt 控件只能在 GUI 线程更新，因此工作线程得到 `DetectionResult` 后，应通过 `pyqtSignal` 发送到界面线程，再调用 `ResultPanel.add_result()`。

建议链路：

```text
抓包线程
→ PacketParser
→ SignatureDetectionService.detect()
→ pyqtSignal(DetectionResult)
→ GUI 线程
→ ResultPanel.add_result()
```

## 8. 错误处理

规则文件及规则对象问题使用：

```python
from src.core.exceptions import RuleError
```

GUI 示例：

```python
try:
    service.reload_rules(path)
except RuleError as exc:
    QMessageBox.warning(window, "规则加载失败", str(exc))
```

传入非 `PacketInfo` 对象时，`detect()` 抛出 `TypeError`，用于尽早发现成员 B 或主程序的接口接线错误。

## 9. 与 DetectionManager 的关系

`SignatureDetectionService` 只负责特征检测，适合成员 E 的独立特征检测页面。系统主流程需要同时运行特征检测和行为检测时，成员 A仍可使用现有：

```python
DetectionManager(SignatureEngine, BehaviorEngine)
```

后续集成时可以让主程序共享服务当前的：

```python
service.engine
```

或者由成员 A建立更高层控制器统一管理特征服务与行为引擎。

## 10. 当前边界

本服务已经解决：

- GUI 与成员 C 模块之间缺少统一入口的问题。
- 规则重新加载后正则缓存没有刷新的问题。
- 内存修改后活动引擎不同步的问题。
- GUI 所需 `DetectionResult` 包装问题。
- 规则加载失败时保留旧活动规则的问题。

仍需其他成员完成：

- 成员 B提供实时 `PacketInfo`。
- 成员 E将模拟检测替换为服务调用。
- 成员 A决定服务在主程序中的生命周期与数据库持久化方式。
- 成员 D提供特征攻击及正常流量样本，用于误报和漏报调优。
