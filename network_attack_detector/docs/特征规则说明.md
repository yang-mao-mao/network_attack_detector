# 特征规则说明

## 1. 文档目的

本文档说明网络攻击检测系统中特征规则的文件格式、字段含义、命名规范、匹配方式、加载校验流程和维护方法。

特征规则由成员 C 负责维护，主要用于检测以下攻击：

- SQL 注入
- XSS
- 命令执行
- WebShell
- 敏感文件访问

规则文件位于：

```text
data/rules/signature_rules.csv
```

系统通过 `RuleLoader` 将 CSV 记录转换为 `SignatureRule` 对象，再由 `SignatureEngine` 对标准化的 `PacketInfo` 数据进行匹配。新增或修改规则不需要修改特征匹配引擎代码。

## 2. 当前规则库概况

当前规则库共包含 45 条特征规则：

| 规则类型 | 数量 | ID 前缀 | category |
|---|---:|---|---|
| SQL 注入 | 12 | `SIG-SQLI-` | `SQL Injection` |
| XSS | 9 | `SIG-XSS-` | `XSS` |
| 命令执行 | 9 | `SIG-CMDI-` | `Command Injection` |
| WebShell | 8 | `SIG-WSHL-` | `WebShell` |
| 敏感文件访问 | 7 | `SIG-SFIL-` | `Suspicious Traffic` |

当前公共枚举 `AttackCategory` 尚未定义独立的“Sensitive File Access”，因此敏感文件访问规则暂时使用 `Suspicious Traffic` 分类，并通过 `SIG-SFIL-` 前缀进行区分。如果后续新增独立枚举，需要同步修改规则文件、统计界面、数据库及报告模块。

## 3. CSV 文件要求

### 3.1 文件编码

规则文件必须使用 UTF-8 编码，可以包含 UTF-8 BOM。加载器以严格模式解码文件，编码错误会转换为 `RuleError`。

### 3.2 完整表头

CSV 必须包含以下 12 个表头：

```csv
rule_id,name,category,level,protocol,match_type,pattern,target_fields,nocase,enabled,description,suggestion
```

要求如下：

- 表头不可缺失。
- 表头名称不可重复。
- 不允许出现没有对应表头的多余列。
- 表头和字段值的首尾空白会在加载时去除。
- CSV 不得为空，也不能只有表头而没有规则记录。
- 包含英文逗号的字段必须使用双引号包围，并按 CSV 规范转义内部双引号。

## 4. 字段说明

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `rule_id` | 字符串 | 是 | 全局唯一的规则标识，遵循统一命名规范。 |
| `name` | 字符串 | 是 | 简短且可读的英文规则名称。 |
| `category` | 枚举字符串 | 是 | 攻击分类，必须与 `AttackCategory.value` 完全一致。 |
| `level` | 枚举字符串 | 是 | 告警等级，必须与 `AlertLevel.value` 完全一致。 |
| `protocol` | 枚举字符串 | 是 | 规则适用的协议。使用 `UNKNOWN` 时表示不限制协议。 |
| `match_type` | 字符串 | 是 | 匹配类型，只允许 `content` 或 `regex`。 |
| `pattern` | 字符串 | 是 | 字符串特征或 Python 正则表达式，不允许为空。 |
| `target_fields` | 分号分隔字符串 | 否 | 需要检查的 `PacketInfo`/`HttpInfo` 字段；为空时默认检查 `payload_text`。 |
| `nocase` | 严格布尔字符串 | 是 | 是否忽略大小写，只允许 `true` 或 `false`。 |
| `enabled` | 严格布尔字符串 | 是 | 是否启用规则，只允许 `true` 或 `false`。 |
| `description` | 字符串 | 否 | 规则命中含义及检测目的。 |
| `suggestion` | 字符串 | 否 | 面向管理员或开发者的修复建议。 |

虽然 `description` 和 `suggestion` 可以为空，但正式规则应完整填写，便于 GUI 展示、告警存储和报告导出。

## 5. 规则 ID 命名规范

规则 ID 统一采用：

```text
SIG-<类型缩写>-<三位序号>
```

例如：

```text
SIG-SQLI-001
SIG-XSS-003
SIG-CMDI-005
SIG-WSHL-002
SIG-SFIL-007
```

类型缩写约定：

| 类型 | 缩写 |
|---|---|
| SQL 注入 | `SQLI` |
| XSS | `XSS` |
| 命令执行 | `CMDI` |
| WebShell | `WSHL` |
| 敏感文件访问 | `SFIL` |

维护要求：

- `rule_id` 在整个规则文件中必须唯一。
- 新规则使用当前类型尚未占用的序号。
- 已发布规则的 ID 不应随意修改或重复利用。
- ID 不得包含首尾空白。
- 特征规则 ID 不应与行为规则 ID 重复。

## 6. 合法枚举值

### 6.1 category

`category` 必须是以下值之一，大小写和空格必须保持一致：

```text
SQL Injection
XSS
Command Injection
WebShell
Brute Force
Port Scan
Malware
Suspicious Traffic
Unknown
```

特征规则当前主要使用 `SQL Injection`、`XSS`、`Command Injection`、`WebShell` 和 `Suspicious Traffic`。

### 6.2 level

```text
Low
Medium
High
Critical
```

建议使用原则：

| 等级 | 建议场景 |
|---|---|
| `Low` | 风险较低、信息性或需要结合其他证据判断。 |
| `Medium` | 可疑探测、弱特征或可能存在一定误报。 |
| `High` | 明确攻击特征，但不一定已获得代码执行能力。 |
| `Critical` | 可能导致远程命令执行、凭据泄露或服务器完全失陷。 |

### 6.3 protocol

```text
TCP
UDP
ICMP
HTTP
HTTPS
ARP
UNKNOWN
```

当前 Web 特征规则通常使用 `HTTP`。`UNKNOWN` 表示不限制协议，可以匹配任意 `PacketInfo.protocol`。

## 7. 匹配类型

### 7.1 content 字符串匹配

`content` 用于固定字符串特征匹配。

示例：

```csv
SIG-SQLI-005,SQLi Information Schema,SQL Injection,High,HTTP,content,information_schema,http.url;http.query;http.body;payload_text,true,true,Detects attempts to enumerate SQL metadata,Use least-privilege database accounts and parameterized queries
```

行为说明：

- `pattern` 不允许为空。
- `nocase=true` 时忽略大小写。
- 告警证据记录数据包中的实际命中文本。
- 适合明确、稳定且不需要复杂变化形式的攻击关键词。

### 7.2 regex 正则匹配

`regex` 使用 Python `re` 语法，可描述空白、编码形式和多种攻击变体。

示例：

```csv
SIG-SQLI-001,SQLi UNION SELECT,SQL Injection,High,HTTP,regex,union\s+(?:all\s+)?select,http.url;http.query;http.body;payload_text,true,true,Detects UNION based SQL injection attempts,Use parameterized queries and validate all untrusted input
```

行为说明：

- `pattern` 不允许为空。
- 规则加载时验证正则能否编译。
- 检测引擎启动时预编译并缓存正则表达式。
- 无效正则会被安全跳过，不会中断其他规则检测。
- `nocase=true` 时使用忽略大小写的正则标志。
- 正则应尽量包含边界或上下文，避免过宽表达式造成误报。

如果正则中含有英文逗号，例如 `{8,}`，整个 `pattern` 字段必须使用 CSV 双引号包围：

```csv
"\b0x[0-9a-f]{8,}\b"
```

## 8. target_fields 白名单

多个目标字段使用英文分号 `;` 分隔，例如：

```text
http.url;http.query;http.body;payload_text
```

当前允许的目标字段如下：

| 字段 | 说明 |
|---|---|
| `payload_text` | 数据包载荷解码后的文本。 |
| `raw_summary` | 原始数据包摘要。 |
| `src_ip` | 源 IP。 |
| `dst_ip` | 目标 IP。 |
| `src_port` | 源端口。 |
| `dst_port` | 目标端口。 |
| `protocol` | 数据包协议。 |
| `http.method` | HTTP 请求方法。 |
| `http.host` | HTTP Host。 |
| `http.url` | 解码后的完整 URL。 |
| `http.path` | 解码后的 URL 路径。 |
| `http.query` | 解码后的查询字符串。 |
| `http.user_agent` | User-Agent。 |
| `http.status_code` | HTTP 响应状态码。 |
| `http.headers` | HTTP 头部字典的文本表示。 |
| `http.body` | 解码后的 HTTP Body。 |

匹配规则：

- 按 `target_fields` 中的顺序逐个检查。
- 同一规则对一个数据包只产生一次告警。
- 命中第一个目标字段后停止检查该规则的其他字段。
- `target_fields` 为空时默认匹配 `payload_text`。
- 不在白名单中的字段会在加载阶段产生 `RuleError`。

## 9. 布尔字段

`nocase` 和 `enabled` 采用严格布尔格式，仅允许：

```text
true
false
```

大小写形式如 `TRUE`、`False` 可以被识别，但正式规则应统一使用小写。以下写法均不允许：

```text
1
0
yes
no
on
off
```

字段含义：

- `nocase=true`：字符串或正则匹配时忽略大小写。
- `nocase=false`：大小写敏感。
- `enabled=true`：规则参与检测。
- `enabled=false`：加载规则但不参与检测。

## 10. 加载、重新加载与校验

### 10.1 初次加载

```python
from src.rules.rule_loader import RuleLoader

loader = RuleLoader()
rules = loader.load_signature_rules("data/rules/signature_rules.csv")
```

返回值：

```python
list[SignatureRule]
```

加载成功后，结果同时保存在：

```python
loader.signature_rules
```

### 10.2 重新加载

```python
rules = loader.reload_signature_rules("data/rules/signature_rules.csv")
```

重新加载会开启新的校验会话，重新统计行号和重复 ID。只有完整文件加载成功后才会替换 `loader.signature_rules`；如果加载失败，上一次成功加载的规则仍会保留。

### 10.3 校验顺序

规则加载过程依次检查：

```text
文件存在性
→ 文件类型
→ UTF-8 编码
→ 空文件
→ CSV 表头
→ CSV 行结构
→ 必填字段
→ 重复 rule_id
→ category/level/protocol 枚举
→ match_type
→ content/regex pattern
→ target_fields
→ nocase/enabled
→ SignatureRule 字段转换
```

## 11. 错误格式

规则相关格式错误统一抛出 `RuleError`。

普通规则错误格式：

```text
Signature rule error [rule_id=<规则ID>, line=<CSV行号>]: <错误原因>
```

示例：

```text
Signature rule error [rule_id=SIG-SQLI-001, line=3]: duplicate rule_id: SIG-SQLI-001
```

```text
Signature rule error [rule_id=SIG-TEST-001, line=4]: invalid regex pattern: missing ), unterminated subpattern
```

表头错误使用：

```text
Signature rule error [rule_id=<header>, line=1]: CSV header missing columns: [...]
```

文件不存在、空文件和编码错误同样会转换为 `RuleError`，调用方无需分别处理 `FileNotFoundError`、`UnicodeDecodeError` 或 `csv.Error`。

## 12. 告警输出

规则命中后，`SignatureEngine.detect(packet)` 返回：

```python
list[Alert]
```

告警包含：

- 攻击类型和告警等级
- 源/目标 IP 和端口
- 数据包协议
- 命中规则 ID 和名称
- 命中证据
- 规则描述和修复建议
- 原始数据包 ID

命中字段和匹配类型记录在 `Alert.extra`：

```python
{
    "matched_field": "http.query",
    "match_type": "regex",
}
```

证据超过 `DEFAULT_PAYLOAD_PREVIEW_LENGTH` 时会被截断，并在末尾添加 `...`。当前默认最大长度为 512 个字符。

系统使用 `packet_id + rule_id` 进行去重，同一规则对同一数据包不会重复产生告警。单条错误规则不会影响其他规则继续检测。

## 13. 添加规则流程

### 13.1 确定攻击特征

优先选择稳定且能代表攻击意图的特征，避免只使用过于常见的短词。例如，命令执行检测应优先匹配命令分隔符与具体命令组合，而不是仅匹配 `id` 或 `cat`。

### 13.2 分配规则 ID

根据类型选择前缀，并使用未占用的三位序号：

```text
SIG-SQLI-013
```

不得修改已有规则的 ID 来腾出序号。

### 13.3 选择匹配类型

- 固定关键词使用 `content`。
- 存在空白变化、可选项或多种写法时使用 `regex`。
- 正则尽量使用边界、上下文和有限长度，避免灾难性回溯。

### 13.4 选择目标字段

根据特征实际出现位置选择最小必要字段集合：

```text
http.url;http.query
```

只有攻击内容可能出现在原始请求中时，再加入：

```text
http.body;payload_text
```

目标字段过多可能增加误报和检测开销。

### 13.5 填写说明和建议

`description` 应说明检测到的攻击行为，`suggestion` 应给出可以执行的防护建议，避免只写“危险请求”或“请检查”等模糊内容。

### 13.6 加载并运行测试

在项目根目录执行：

```powershell
python -m pytest -q
python main.py --self-check
```

至少准备：

- 一条能够命中的攻击样本。
- 一条不应命中的正常样本。
- 大小写或编码变化样本。
- 针对 regex 的边界输入。

### 13.7 真实流量调优

使用成员 D 提供的攻击脚本或 pcap 样本检查：

- 是否存在漏报。
- 是否存在正常请求误报。
- 命中字段是否正确。
- 告警等级是否合理。
- 证据是否足以解释告警。

调整规则后应重新运行完整测试。

## 14. 规则管理接口

当前 `RuleManager` 提供纯内存管理接口：

```python
list_rules()
get_rule(rule_id)
add_rule(rule)
remove_rule(rule_id)
update_rule(rule)
replace_signature_rules(rules)
enable_rule(rule_id)
disable_rule(rule_id)
```

注意：当前规则管理只修改内存，不会自动写回 CSV 或 SQLite。持久化、GUI 操作和运行中检测引擎刷新需要与成员 A、E 的模块进行集成。

## 15. 维护注意事项

- 不要在未运行测试的情况下直接提交新规则。
- 不要使用重复 ID，也不要重复利用已删除规则的 ID。
- 不要使用未经约束的宽泛正则。
- 不要把正常业务中高频出现的短字符串直接设为高等级规则。
- 修改公共枚举或目标字段前，应与成员 A、B、E 协调。
- 修改规则后应同时检查正例命中率和正常流量误报率。
- `__pycache__`、`.pyc` 和 `.pytest_cache` 不属于规则交付物，不应提交到版本库。
